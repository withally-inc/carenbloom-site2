ATS / Job Distribution Decision

Updated June 9, 2026 · Source doc: ATS_DISTRIBUTION_RESEARCH.md
Decision surface

Executive answer

Start distribution-first. Pilot JobTarget or eQuestXpress before replacing the recruiting workflow. The current Care & Bloom form and Notion intake already cover most workflow needs; the missing capability is reliable multi-board posting with source tracking and field parity.

Why this is the right starting point

  • The core problem is distribution, not workflow. You already have a role-aware application form and Notion intake.
  • ATS-first creates migration drag. JazzHR, Workable, and Ashby are useful fallback tests, but they may push candidates into hosted forms before proving board reach.
  • DIY is helpful but incomplete. Google Jobs schema and a feed are worth building, but LinkedIn, Indeed, and ZipRecruiter posting automation is partner/approval-gated.
  • Field parity is the guardrail. Native board apply is acceptable only if resume, phone, location, timezone, monthly income, role questions, intro video, and source arrive cleanly.
  • Pricing unknown is not free. No public pricing means budget risk until a quote or cart proves the total under $1K/month.

Pricing guardrail

Important correction

Unknown pricing is not free

If a vendor does not publish pricing, treat it as quote-only or budget risk until a quote or cart proves it fits. Missing pricing should never be interpreted as free.

Distributor nuance

No platform fee is not no cost

JobTarget says there is no additional platform fee, but postings, campaigns, premium products, paid board inventory, and sometimes delivery/processing may still cost money. eQuestXpress says no setup/minimums and pay-as-you-go, which still implies per-use spend.

Decision paths

Preferred

Distribution-first

Pilot JobTarget or eQuestXpress with 2-3 roles. Keep Care & Bloom form/Notion if custom apply URLs and source parameters work.

Fallback

ATS-owned workflow

Use JazzHR or Workable if distribution-first tools cannot preserve apply flow. Use Ashby if workflow/data rigor becomes more important than board breadth.

Not enough alone

DIY posting

Build schema/feed/source tracking, but do not expect direct APIs to cover LinkedIn, Indeed, and ZipRecruiter reliably.

Board coverage reality

Core boards

  • Google Jobs: easiest direct path via structured data.
  • Indeed: feed/Apply exist, but approval and quality checks matter.
  • LinkedIn: API access is partner-gated.
  • ZipRecruiter: partner-oriented API/webhook route.

Remote + Asia gap

Public docs do not prove coverage for OnlineJobs.ph, Kalibrr, Bossjob, Glints, or JobStreet across the shortlist. This is a top demo question for JobTarget, eQuest, JOIN, Broadbean, and VONQ.

Shared foundation before vendor work

Build first

Source and file plumbing

Add source, utm_source, utm_medium, utm_campaign, utm_vendor, referrer, and landing URL to Notion. Replace filename-only resume handling with file uploads or resume URLs.

Build first

Vendor-ready jobs feed

Create stable JSON/XML with external job id, title, slug, public role URL, apply URL, remote/location, description, department, date posted, close date, and active/closed status.

JobTargetMedium-low work

Best model: keep Care & Bloom as the application source of truth. JobTarget imports jobs or receives manually-created jobs, then sends each board posting to a role-specific apply URL.

Steps

  1. Build a stable jobs feed or importer-friendly careers page.
  2. Add source/referrer/UTM persistence in Notion.
  3. Fix resume upload or resume URL capture.
  4. Confirm which boards support redirect apply vs JobTarget Easy Apply/Candidate Manager.
  5. If a key board forces hosted apply, map required fields and export back into Notion.

Difficulty flags

  • Importer paths can be daily/nightly, so edits may not appear instantly.
  • Source parameters must survive board and JobTarget redirects.
  • Free boards may still have delivery/posting fees in some partner flows.

Decision check

Pick it if custom apply URLs work on the actual board list. Do not pick it only because the platform fee is zero; posting and campaign costs still apply.

eQuestXpressMedium work

Best model: keep Care & Bloom as application source of truth. Use AutoPost, career-site scrape, or feed import, with eQuest source codes passed into the apply URL.

Steps

  1. Create a feed or make job pages structured enough for AutoPost.
  2. Add source-code passthrough into apply URLs.
  3. Add completed-application tracking, likely a thank-you page/event or apply pixel, if available.
  4. Persist source/referrer/UTM data in Notion.
  5. Confirm board list for US remote, Europe, and Philippines/Asia.

Difficulty flags

  • Exact scrape/feed setup needs vendor configuration.
  • Completed-apply attribution may need a new completion page or event.
  • Public docs do not fully expose board-by-board apply behavior or total cost.

Decision check

Pick it if it proves global board reach plus clean redirect/source tracking. Treat it as pay-as-you-go with paid board/cart spend, not free.

JOINMedium/high work

Best model: trial it as a hybrid. First decide whether JOIN collects candidates, or whether Care & Bloom keeps the form and pushes applications into JOIN.

Steps if JOIN owns applications

  1. Configure JOIN jobs/forms and screening questions.
  2. Map Care & Bloom required fields into JOIN questions.
  3. Use JOIN export, API, or automation into Notion.
  4. Represent role-specific workflows inside JOIN.

Steps if Care & Bloom form stays

  1. Confirm JOIN can use a custom apply URL for multiposted jobs.
  2. If not, bridge Care & Bloom applications into JOIN using POST /v2/applications.
  3. Map resumes/documents into JOIN attachment fields.
  4. Store JOIN application ids in Notion to prevent duplicates.

Difficulty flags

  • Public API docs prove application endpoints, not custom apply URL behavior for every board.
  • API/export/custom screening appear tied to higher plan functionality.
  • The 15 active-job cap may matter if many roles are live simultaneously.

Decision check

Trial it only if 15 active jobs is enough and the apply path is acceptable. If JOIN must own workflow, compare directly against JazzHR first.

JazzHRMedium work

Best model: budget ATS fallback that can plausibly preserve the Care & Bloom form through JazzHR Apply API. If JazzHR owns workflow, use Candidate Export webhooks back to Notion.

Steps

  1. Create each role in JazzHR and map site role slugs to JazzHR jobBoardCode.
  2. Fetch JazzHR questions per role and map them to Care & Bloom form fields.
  3. Update /api/applications or add a server integration endpoint that submits to JazzHR.
  4. Add real resume upload/base64 handling inside JazzHR limits.
  5. Manage Apply API authentication and token refresh.
  6. Optionally sync stage changes back to Notion through Candidate Export webhooks.

Difficulty flags

  • Apply API token lifetime is short, so backend refresh/cache is needed.
  • Resume file limits and MIME rules are stricter than the current filename-only flow.
  • Role-specific JazzHR questions can change, so mapping needs maintenance.

Decision check

This is the best "budget ATS but keep our form" candidate. The demo should include a real API application submit test.

WorkableMedium/high work

Best model: Workable owns application intake and recruiting workflow. Care & Bloom can list jobs, but likely routes applications to Workable forms or job shortlinks.

Steps

  1. Configure jobs, custom application forms, stages, and source tracking in Workable.
  2. Use Workable shortlinks/tracking links for distributed postings.
  3. Optionally use the Workable careers page API to list jobs on the Care & Bloom site.
  4. Map required fields into Workable custom questions.
  5. Sync to Notion through Workable API/webhooks if Notion remains needed.

Difficulty flags

  • Public docs show job listing and candidate retrieval better than custom-apply submission.
  • Source tracking can become "Unknown" in some visitor/session cases.
  • Hosted forms may be a larger workflow change than desired.

Decision check

Demo it if workflow may move into the ATS. Do not pick it purely for distribution unless custom apply URL/source handling is proven.

AshbyHigh work

Best model: Ashby becomes the recruiting source of truth. It can support a custom Care & Bloom apply experience, but this is a real API integration, not just multiposting setup.

Steps

  1. Create jobs and application forms in Ashby.
  2. Map current role slugs to Ashby job posting ids.
  3. Build a server-side proxy; do not call Ashby's private API directly from the browser.
  4. Use jobPosting.list/jobPosting.info to render jobs/forms, or map existing fields manually.
  5. Submit applications with applicationForm.submit, including files via multipart or upload handles.
  6. Pass utmData so source/campaign tracking is preserved.
  7. Sync Ashby back into Notion only if Notion remains a reporting layer.

Difficulty flags

  • Highest architecture lift because Ashby should own workflow to justify migration.
  • Custom form field paths/connectors need careful mapping.
  • File upload and application submission require secure backend work.
  • Pricing remains quote-needed.

Decision check

Choose Ashby if workflow/data rigor matters more than broad marketplace distribution. Do not choose it if the only job is cheap one-click posting to many niche boards.
